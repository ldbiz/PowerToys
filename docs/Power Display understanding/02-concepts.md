# Power Display concepts

| Concept | Meaning in this module | Where it appears | Why it matters for brightness debugging |
|---|---|---|---|
| Power Display module lifecycle | The runner loads a native DLL, which starts/stops the WinUI app process while the module is enabled. | `PowerDisplayModuleInterface/dllmain.cpp`, `PowerDisplayProcessManager.cpp`, `PowerDisplay/Program.cs`, `App.xaml.cs` | Brightness control only exists in the WinUI process after startup, crash recovery, GPO checks, and event/pipe setup complete. |
| Sidecar WinUI process | `PowerToys.PowerDisplay.exe` owns the flyout, tray icon, monitor discovery, and hardware control. | `Program.cs`, `App.xaml.cs`, `MainWindow.xaml.cs`, `MainViewModel*.cs` | If the process is not running, monitor enumeration and brightness writes do not occur even if the runner module is loaded. |
| Monitor discovery | The app builds active display inventory, routes WMI-claimed displays to WMI, and routes the rest to DDC/CI. | `MonitorManager.cs`, `DisplayConfigInventory.cs`, `WmiController.cs`, `DdcCiController.cs` | A VDI display may appear in Windows but fail one of these module-specific discovery stages. |
| DisplayConfig inventory | Active display paths are read with QueryDisplayConfig and device info calls before controller discovery. | `DisplayConfigInventory.cs`, native structures under `PowerDisplay.Lib/Drivers/NativeStructures/` | This is the first mapping between “Windows has a display” and “Power Display has a monitor candidate.” Empty or incomplete inventory aborts later discovery. |
| WMI brightness path | Internal-panel brightness is discovered/read/written through `root\WMI` classes `WmiMonitorBrightness` and `WmiMonitorBrightnessMethods`. | `WmiController.cs` | If a display is visible in Windows Settings but absent from these WMI classes, Power Display does not treat it as WMI-controllable. |
| DDC/CI brightness path | External displays are discovered through logical monitor handles and physical monitor handles; brightness uses VCP code `0x10`. | `DdcCiController.cs`, `DdcCiNative.cs`, `PhysicalMonitorHandleManager.cs` | VDI or virtual display stacks may not supply the physical-monitor/capability path Power Display expects. The repo documents this dependency but not external VDI behavior. |
| Brightness read | Discovery initializes `Monitor.CurrentBrightness` from WMI current brightness or DDC VCP `0x10` when supported. | `WmiController.GetBrightnessAsync`, DDC initialization in `DdcCiController.cs`, `Monitor.cs` | A monitor can be discovered but show no brightness slider if the capability flag is not set or current value cannot be initialized. |
| Brightness write | UI sliders call `MonitorViewModel.SetBrightnessAsync`, which delegates to `MonitorManager.SetBrightnessAsync` and then to the monitor’s selected controller. | `MonitorViewModel.cs`, `MonitorManager.cs`, `WmiController.cs`, `DdcCiController.cs` | Diagnosing write failure requires knowing the monitor's `CommunicationMethod`, ID, and controller result. |
| Display identity/mapping | The canonical `Monitor.Id` is derived from QueryDisplayConfig device paths; WMI instance names are transformed to the same shape for pairing. | `MonitorIdentity.cs`, `DisplayConfigInventory.cs`, `WmiController.cs`, `MonitorManager.cs` | Identity mismatches can make a WMI brightness instance look unrelated to an active display. |
| Physical vs logical display representation | Logical display paths/GDI names come from DisplayConfig; DDC also uses logical `hMonitor` handles and physical monitor handles. | `MonitorDisplayInfo.cs`, `DdcCiController.cs`, `PhysicalMonitorHandleManager.cs`, `Monitor.cs` | “Display visible in Windows” is not equivalent in this code to “physical monitor handle and VCP brightness available.” |
| Monitor capability detection | DDC monitors depend on MCCS capability strings or max-compat probing; WMI monitors imply brightness support when WMI pairing succeeds. | `MccsCapabilitiesParser.cs`, `DdcCiController.cs`, `MonitorCapabilities.cs`, `Monitor.cs` | Unsupported/failed capability detection can remove the brightness UI without implying no display exists. |
| Settings model | Power Display settings persist monitor entries, visibility, linked brightness, max compatibility, restore-on-startup, profiles, and hotkey. | `PowerDisplaySettings.cs`, `PowerDisplayProperties.cs`, `MainViewModel.Settings.cs`, `PowerDisplayViewModel.cs` | Settings can hide monitors, enable restore writes, alter DDC compatibility behavior, and control what the UI exposes. |
| UI/backend boundary | The UI is in-process with `MonitorManager`; the Settings UI communicates via settings files and named events rather than direct monitor-control calls. | `MainViewModel.Settings.cs`, `PowerDisplayViewModel.cs`, `App.xaml.cs`, `NamedPipeProcessor.cs` | Settings changes can trigger refresh or profile application, but brightness slider writes are app-to-controller calls inside Power Display. |
| IPC and signaling | Runner/module DLL uses named pipe/events; Settings UI and Power Display use named events plus settings reloads. | `PowerDisplayProcessManager.cpp`, `NamedPipeProcessor.cs`, `App.xaml.cs`, `PowerDisplayViewModel.cs` | Failure to receive events affects refresh/settings behavior, not the underlying monitor API path directly. |
| Error handling/reporting | Controllers return `MonitorOperationResult`, log warnings/errors, and often continue with empty monitor lists on discovery failures. | `MonitorManager.cs`, `WmiController.cs`, `DdcCiController.cs`, `MonitorViewModel.cs`, `DisplayConfigInventory.cs` | VDI symptoms may appear as empty discovery, missing capabilities, warning logs, or slider-write failures rather than a hard exception. |
| Test doubles/mocks | Unit tests mostly target pure logic and file/state behavior; direct OS monitor APIs are not heavily mocked in visible tests. | `PowerDisplay.Lib.UnitTests/` | There is limited direct test evidence for virtual displays, remote sessions, physical handle absence, or WMI provider differences. |
| VDI/virtual display relevance | The code distinguishes active display inventory, WMI brightness exposure, and DDC/CI physical-monitor capability. | Discovery and controller files above | This split is the concrete repo-supported reason to inspect API-path differences in VDI, without assuming which path fails. |
